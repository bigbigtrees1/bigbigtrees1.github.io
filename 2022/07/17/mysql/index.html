数据库介绍

1. 数据库概念

数据库，就是存放数据的仓库。
数据库(DataBase,简称DB)长期存储在计算机内部有结构的、大量的、共享的数据集合。

  • 长期存储:持久存储
  • 有结构
    • 类型:数据库不仅可以存放数据，而且存放的数据还是有类型的
    • 关系:存储数据与数据之间的关系
  • 大量:大多数数据库都是文件系统的，也就是说存储在数据库中的数据实际上就是存储在磁盘的文件中。
  • 共享:多个应用程序可以通过数据库实现数据的共享。

2.关系型数据库与非关系型数据库

  • 关系型数据库

采用了关系模型来组织数据的存储，以行和列的形式存储数据并记录数据与数据之间的关系————将数据存储在表格中，可以通过建立表格与表格之间的关联来维护数据与数据之间的关系。
学生信息————学生表
班级信息————班级表

  • 非关系型数据库

采用键值对的模式来存储数据，只完成数据的记录，不会记录数据与数据之间的关系。
在非关系型数据库中基于其特定的存储结构来解决一些大数据应用的难题。
NoSQL(Not only SQL)数据库来指代非关系型数据库

8.SQL命令

8.1 SQL概述

SQL(Structured Query Language) 结构化查询语言，用于存取，查询，更新数据以及管理关系型数据库系统。

8.2 SQL发展

  • SQL实在1981年由IBM公司，一经退出基于其间接的语法在数据库中得到了广泛的运用，成为主流数据库的通用规范。
  • SQL由ANSI组织确定规范
  • 在不同的数据库产品中遵守SQL的通用规范，但是也对SQL有一些不同的改进，形成了一些数据库的专有指令
    • MySQL:limit
    • SQLServer:top
    • Oracle:rownum

8.3 SQL的分类

根据SQL指令完成的数据库操作不同，可以将SQL指令分为四类:

  • DDL Data Definition Language 数据定义语言
    • 用于完成对数据库对象(数据库、数据表、试图、索引等)的创建、删除、修改
  • DML Data Mainipulation Language 数据操作语言
    • 用于完成对数据表中的数据的添加、删除、修改操作
    • 添加:将数据存储到数据表中
    • 删除:将数据从数据表移除
    • 修改:对数据表中的数据进行修改
  • DQL Data Query Language 数据查询语言
    • 用于将数据表中的数据查询出来
  • DCL Data Control Language 数据控制语言
    • 用于完成事务管理等控制操作

8.4 SQL基本语法

  • 每条SQL指令结束之后都以;结束

在创建数据表的时候，指定的对数据表的列的数据限制性的要求(对表的列中的数据进行限制)
为什么要给表中的列添加约束呢?

  • 保证数据的有效性
  • 保证数据的完整性
  • 保证数据的正确性

字段常见的约束有哪些呢?

  • 非空约束: not null : 限制此列的值必须提供，不能为null

8.9.2 where子句

在删除、修改及查询的语句后都可以添加where子句(条件)，用于筛选满足特定的条件的数据进行删除、修改和查询操作。

  • delete from <tableName> where <conditions>;

  • update <tableName> set <columnName> = value <where conditions>;

  • select <colnumName1, colnumName2, colnumName3>from <tableName> where <conditions>;

  • 条件

  • 多条件查询

在where子句中，可以将多个条件通过逻辑预运算(and or not)进行连接，通过多个条件来筛选要操作的的数据。

and:筛选多个条件同时满足
select * from students where sut_age > 20 and std_gender = ‘女’;

or：筛选多个条件中至少满足一个条件的记录
select * from students where sut_age > 20 or std_gender = ‘女’;

not:取反
select * from students where stu_age not between 18 and 20;

8.9.3 LIKE子句

在where子句的条件中，我们可以使用LIKE关键字来实现模糊查询

  • 语法

select * from <tableName> where <columnName> like ‘reg’;

  • 在like关键字后的reg表达式中%表示任意多个字符

  • 示例
    select * from students where stu_num like ‘%o%’;
    注释:%o%表示字符串里面包含o就行，o必须被包含，前后包含(应该)

  • _表示任意一个字符 比如_o，只能找ao这些o前面只有一个的

  • 设置查询的列

  • 计算列

    select <distinct> <stu_age> from <stus>;

8.9.5 排序-order by

将查询到的满足条件的记录按照指定的列的值升序或者降序排序

  • 语法
    select * from <tableName> where <conditions> order by <columnName> asc | desc;

    • order by columnName 表示将查询到的结果按照指定的列排序
      • asc 表示升序(默认)
      • desc 表示降序
    • 注意:where和后面的条件conditions可以不写

  • 示例

    • 单字段排序
      • select * from stus where stu_age > 22 order by stu_gender;
      • select * from stus order by stu_gender desc;
    • 多字段排序 先满足第一个排序规则再按照后面的满足
      • select * from stus where stu_age > 10 order by stu_gender desc , stu_age desc;

8.9.8 分组查询-group by

分组–就是将数组表中的记录按照指定的类进行分组

  • 语法

select 分组字段/聚合函数 from 表名 [where 条件] group by 分组字段 [having 条件] [order by 排序];

  • []代表可有可无

  • select 后使用*显示对查询的结果进行分组之后，显示每组的第一条记录(这种操作通常是无意义的)

  • select后通常显示分组字段和聚合函数(对分组后的数据进行统计、求和、平均值等)

  • 语句执行顺序:

    • 1.先根据while条件从数据库查询数据
    • 2.group by 对查询数据进行分组
    • 3.执行having对分组后的数据进行筛选
    • 4.最后根据order by 进行排序

  • 注意: 没有group by 不能又having

  • 示例1

先对查询的学生信息按照性别进行分组(分成了男和女两组)，然后再分别统计每组学生的个数

  • select stu_gender, count(stu_num) from stus group by stu_gender;
  • 示例2

先对学生按年龄进行分组，然后统计各组的学生数然后进行排序

  • select stu_age, count(stu_num) from stus group by stu_age order by stu_age;
  • select stu_age, count(stu_num) from stus group by stu_age order by stu_age;
  • 示例3

按年龄进行分组，然后分别统计每组的人数，再筛选当前组人数>1的组显示出来

  • select stu_age, count(stu_num) from stus group by stu_age having count(stu_num) >= 1 order by stu_age;
  • 示例4

查询性别为男的学生，然后根据年龄进行分组，然后分别统计每组的人数，再筛选当前组人数>1的组显示出来

  • select stu_age, count(stu_num) from stus where stu_gender=’女’ group by stu_age having count(stu_num) >= 1 order by stu_age;
8.9.7 分页查询-limit

当数据表中的记录比较多的时候，如果一次性查询出来显示给用户，用户的可读性/体验性不太好，因此我们可以将这些数据分页进行展示。

  • 语法

select … from … where … limit param1, param2

  • param1 int类型, 表示查询语句的结果中的第一条数据的索引(索引从0开始)
  • param2 int类型，表示获取的查询记录的条数(如果剩下的数据条数 < param2则返回剩下的所有数据)
  • 案例:对数据表中的学生信息进行分页显示，总共有10条数据，我们每页显示5条

总记录数 count 6条
每页显示 pageSize 3
总页数 pageCount = count % pageSize == 0 ? count % pageSize : count % pageSize + 1

  • 查询第一页:
    select * from stus where … limit 0,3;
  • 查询第二页:
    select * from stus where … limit 3,3;
  • 查询第三页:
    select * from stus where … limit 6,3;
  • 查询第四页:
    select * from stus where … limit 9,3;
  • 如果在一张数据表中
    pageNum表示查询的页码
    pageSize表示每页显示的条数:
    通用分页语句如下：
    select * from <tableName> where limit (pageName-1)*pageSize, pageSize;

8.10 数据表的关联关系

8.10.1 关联关系介绍

MySQL是一个关系型数据库，不仅可以存储数据，还可以维护数据与数据之间的关系–通过在数据表中添加字段建立外键约束

  • 数据与数据之间的关联关系分为四种:
    • 一对一关联
    • 一对多关联
    • 多对一关联
    • 多对多关联
8.10.2 一对一关联介绍

人–身份证
学生–学籍
用户–用户详情

  • 方案1:主键关联–两张数据表中主键相同的数据为相互对应的数据
  • 方案2:唯一外键–在任意一张表中添加一个字段添加外键约束与另一张表主键关联，并且将外键列添加唯一约束
8.10.3 一对多与多对一

班级–学生 (一对多) 一个班级包含多个学生
学生–班级 (多对一) 多个学生术语同一个班级

图书–分类 商品–商品类别

  • 方案:在多的一段添加外键，与一的一段主键进行关联
8.10.4 多对多关联

学生–课程 一个学生可以选择多门课，一门课程也可以由多个学生选择
会员–社团 一个会员可以参加多个社团、一个社团也可以招纳多个会员

  • 方案:额外创建一张关系表来维护多对多关联–在关系表中定义两个外键，分别与两个数据表的主键进行关联
8.10.5 外键约束

外键约束–将一个列添加外键约束与另一张表的主键(唯一列)进行关联之后，这个外键约束的列添加的数据，必须要在关联的主键字段中存在

案例:学生表与班级表(在学生表添加外键与班级表的主键进行关联)

  • 1.先创建班级表，

    • create table classes(
      class_id int primary key auto_increment,
      class_name varchar(40) not null unique,
      class_remark varchar(200)

  • 2.创建学生表 (在学生表中添加外键与班级表的主键进行关联)

      FK_STUDENTS_CLASSES是外键约束名称

      • create table students(
        stu_num char(8) primary key,
        stu_name varchar(20) not null,
        stu_gender char(2) not null,
        stu_age int not null,
        cid int,

    • ② 创建表的时候，先不添加外键

      • create table students(

  • 4.向学生表中添加学生信息

8.10.6 外键约束-级联

当学生表中存在学生信息关联班级表的某条记录时，我们就不能对班级表的这条记录进行修改ID、删除操作

如果一定要修改如何实现

    1. 将引用Java2014班级id的学生记录中的cid修改为NULL
    1. 再修改班级信息表中Java2014记录的class_id
    1. 将学生表中cid设置为null的记录的cid重新修改为java2014这个班级的新id

我们可以使用级联操作来实现:

  • 1.在添加外键时，设置级联修改和级联删除
    删除原有的外键
    alter table students drop foreign key FK_STUDENTS_CLASSES;
    重新添加外键，并设置级联修改和级联删除
    alter table students add constraint FK_STUDENTS_CLASSES foreign key(cid) references classes(class_id) on update cascade on delete cascade;

  • 2.测试级联代码

    删除class_id=1的班级，会删除students中所有引用class_id=1的学生

8.11.2 内连接INNER JOIN
  • 语法

select … from <tableName1> inner join <tableName2>;

8.11.2.1 笛卡尔积
  • 笛卡尔积(A集合&B集合):使用A中的每个记录依次关联B中的每个记录，笛卡尔积的总数=A总数*B总数
  • 如果直接执行select … from <tableName1> inner join <tableName2>;会获取两种数据表中的笛卡尔积(依次使用tableName1表中的每一条记录去匹配tableName2中的每条数据)
8.11.2.2 内连接条件

两张表使用inner join连接查询之后产生的笛卡尔积数据中很多数据都是无意义的，我们如何消除无意义的数据呢? – 添加两张进行连接查询时的条件

  • ①.第一种方法
    • select * from classes INNER JOIN students where students.cid= classes.class_id;
    • 效率低下，先生成笛卡尔积再从笛卡尔积中过滤数据
  • ②.第二种方法
    • select * from classes INNER JOIN students on students.cid= classes.class_id;
    • 先判断连接条件是否成立，如果成立两张表的数据进行组合
    • on后面还可以加where来筛选条件
    • 结果:只获取两张表中匹配条件成立的数据，任何一张表在另一张表中如果没有找到对应的匹配则不会出现在查询结果中(列如:小红和小明没有对应的班级信息，Java2016和Python2016没有对应的学生)
8.11.3 左连接

需求:请查询出所有的学生信息，如果学生有对应的班级信息，则讲对应的班级信息也查询出来

  • 左连接:显示左表中的所有数据，如果在右表中存在与左表记录满足匹配条件的数据，则进行匹配;如果右表中不存在匹配数据，则显示Null

  • 语法

  • select * from <leftTable> LEFT JOIN <rightTable> ON students.cid= classes.class_id;
    • select * from classes LEFT JOIN students on 匹配条件 [where 条件];
    • 注意:<leftTable>和<rightTable>互换可能不一样，因为<leftTable>是显示所有数据，<rightTable>是只显示与左表有关的，无关的为NULL
8.11.4 右连接
  • 语法
  • select * from <rightTable> RIGHT JOIN <leftTable> ON students.cid= classes.class_id;
8.11.5 数据表别名

如果在连接查询的多张表中存在相同名字的字段，我们可以使用<表名.字段名>来进行区分，如果表名太长则不便于SQL语句的编写，我们可以使用数据表别名

  • 使用示例：
  • select s.*,c.class_name
    from students s
    LEFT JOIN classes c
    ON s.cid= c.class_id;
  • 注意MySQL语句执行顺序
8.11.5 子查询/嵌套查询

子查询-先进性一次查询，第一查询的结果作为第二次查询的源/条件(第二次查询是基于第一次查询的结果来进行的)
注意:如果子查询返回的结果是单个值，条件可以直接使用关系符(=, !=, …)
如果子查询返回的结果是多个值(单列多行)，条件使用就不使用=，使用IN(使用查询结果里面的)，NOT IN(使用不在查询结果里面的)

8.11.5.1 子查询返回单个值-单行单列
  • 案例1: 查询班级名称为Java2104班级中的学生信息(只知道班级名称，而不知道班级id)
    • 传统的方式：
      • ①.查询java2104班的班级编号
      • select class_id from classes where class_name = ‘Java2104’;
      • ②.查询此班级编号下的学生信息
      • select * from students where cid = 1;
    • 子查询:
      • select * from students where cid = (select class_id from classes where class_name = ‘Java2104’);
      • 先执行括号里面的，再把返回的值给外面的执行
8.11.5.2 放回多个值-多行单列
  • 案例2:查询所有Java班级中的学生信息

    • 传统的方式:

      • ①.查询所有Java班的班级编号
      • select class_id from classes where class_name LIKE ‘Java%’;
      • ②.查询这些班级编号中的学生西信息(union 将多个查询语句的结果整合在一起)
      • select * from students where cid = 1
        UNION
        select * from students where cid = 2
        UNION
        select * from students where cid = 3;

    • 子查询:

      • select * from students where cid IN (select class_id from classes where class_name LIKE ‘Java%’);
8.11.5.3 返回多个值-多行多列
  • 案例3:查询cid=1的班级中性别为男的学生信息
    • 传统的方式:
      • select * from students where cid =1 and stu_gender=’男’;
    • 子查询:先查询cid=1班级中的所有学生信息，将这些信息作为一个整体虚拟表(多行多列)，再基于这个虚拟表查询性别为男的学生信息，(‘虚拟表需要别名’)
      • select * from (select * from students where cid =1) t where t.stu_gender=’男’;

8.12 存储过程

8.12.1 存储过程介绍
8.12.1.1 SQL指令执行过程

从SQL指令执行的流程中我们分析存在的问题:
1.如果我们需要重复多次执行相同的SQL，SQL执行需要通过连接传递到MySQL,并且需要经过编译和执行的步骤;
2.如果我们需要连续执行多个SQL指令，并且第二个SQL指令需要使用第一个SQL指令执行的结果作为参数;

8.12.1.2 存储过程的介绍

将能够完成特定功能的SQL指令进行封装(SQL指令集)，编译之后存储在数据库服务器上，并且为之取一个名字，客户端可以通过名字直接调用这个SQL指令集，获取执行结果

  • 优点:
    • ①.SQL指令无需客户端编写，通过网络传送，可以节省网络开销，同时避免SQL指令在网络传输过程中被而已篡改保证安全性
    • ②.存储过程经过编译创建并保存在数据库中，执行过程无需重复的进行编译操作，对SQL的执行过程进行了性能上的提升
    • ③.存储过程中多个SQL指令之间存在逻辑关系，支持流程控制语句(分之、循环)，可以实现更为复杂的业务
  • 缺点：
    • ①.存储过程是根据不同的数据库进行编译、创建并存储在数据库中，当我们需要切换到其他的数据库产品时，需要重新编写针对于新的数据库的存储过程
    • ②.存储过程受限于数据库产品，如果需要高性能的优化会成为一个问题
    • ③.在互联网项目中，如果需要数据库的高(连接)并发访问，使用存储过程会增加数据库的连接执行时间(因为我们将复杂的业务交给了数据库进行处理)

8.13 触发器

8.13.1 触发器的介绍

触发器，就是一种特殊的存储过程。触发器和存储过程一样是一个能够完成特定、存储在数据库服务器上的SQL片段，但是触发器无需调用，当对数据表中的数据执行DML操作时会自动出发这个SQL片段的执行，而无需手动调用。

在MySQL，只有执行insert/delete/update才能触发触发器的执行。

8.13.4 触发器使用总结
8.13.4.1 触发器优点
  • 触发器是自动执行的，当对触发器相关的表执行相应的DML操作时立即执行
  • 触发器可以实现表中的数据级联操作(关联操作)，有利于保证数据的完整性
  • 触发器可以对DML操作的数据进行更为复杂的合法性校验
8.13.4.2 触发器缺点
  • 使用触发器实现的业务逻辑如果出现问题将难以定位，后期维护困难
  • 大量使用触发器容器导致代码结构杂乱，增加了程序的复杂性
  • 当触发器操作的数据量比较大时，执行效率会大大降低
8.13.4.3 使用建议
  • 在互联网项目中，应避免使用触发器
  • 对于并发量不大的项目可以选择使用存储过程，但是在互联网中不提倡使用存储过程(原因:存储过程时将实现业务逻辑交给数据库处理，一则增加了数据库的负载，二则不利于数据库的迁移)

8.14 视图

8.14.1 视图的概念

视图，就是由数据库中一张表或者多张表根据特定的条件查询出的数据结构成得虚拟表

8.14.2 视图的作用
  • 安全性:如果我们直接将数据表授权给用户操作，那么用户可以CRUD数据表中所有数据，加入我们想要对数据表中得部分数据进行保护，可以将公开的数据生成视图;用户通过查询视图可以获取数据表中公开的数据，从而达到将数据表中的部分数据对用户隐藏。
  • 简单性:如果我们需要查询的数据来源于多张数据表，可以使用多表连接查询来实现;我们通过视图将这些连表查询的结果对用户开放，用户则可以通过查询视图获取多表数据，操作更便捷。
8.14.3 创建视图
8.14.2.1 语法

create view 表名
As
查询结果

8.14.2.2 实例

– 创建视图1 将学生表中性别为男的学生生成一个视图
create view view_test1
AS
select * from students where stu_gender = ‘男’;
– 查询视图
select * from view_test1;

– 创建视图2 查询学生借书信息(学生名、图书名、借书数量)
create view view_test2

from books b inner join records r inner join students s
on b.book_id = r.bid and r.snum=s.stu_num;
– 查询视图
select * from view_test2;

8.14.4 视图数据的特性
  • 视图是虚拟表，查询视图的数据是来源于数据表的。当对视图进行操作时，对原数据表中的数据是否有影响呢?
  • 查询操作:如果在数据表中添加了新的数据，而且这个数据满足创建视图时查询语句的条件，通过查询视图也可以查询出新增的数据;当删除原表中满足查询条件的数据时，也会从视图中删除。
  • 新增数据:如果在视图中新增数据，数据会被添加到原数据表
  • 删除数据:如果从视图删除数据，数据也将从原表中删除
  • 修改操作:如果通过视图修改数据，则也将修改原数据表中的数据
  • 注意:对复杂查询简化，并且不会对数据修改的情况下可以使用视图，以上对于单表适用，多表联合查询不推荐使用上诉操作。

8.15 索引

数据库是用来存储数据，在互联网应用中数据库中存储的数据可能会很多(大数据),数据表中数据的查询速度会随着数据量的增长逐渐变慢，从而导致响应用户请求的速度变慢–用户体验差，我们如何提高数据的查询效率?

8.15.1 索引的介绍
  • 索引，就是用来提高数据表中数据的查询效率的。

索引，就是将数据表中某一列/某几列的值取出来构造成便于查找的结构进行存储，生成数据表的目录。当我们进行数据查询的时候，则现在目录中进行查找得到对应的数据抵制，然后再到数据表中根据地址快速的获取数据记录，避免全表扫描。

8.15.2 索引的分类

  • MySQL中的索引，根据创建索引的列的不同，可以分为:

    • 主键索引:在数据表的主键字段创建的索引，这个字段必须被primary key修饰，每张表只能有一个主键
    • 唯一索引:在数据表的唯一列创建的索引(unique)，此列的所有值只能出现一次，可以为NULL
    • 普通索引:在普通字段上创建的索引，没有唯一性的限制
    • 组合索引:两个及以上字段联合起来创建的索引

  • 说明:

    • 1.在创建数据表时，将字段声明为主键，会自动在主键字段创建为主键索引;
    • 2.在创建数据表时，将字段声明为唯一键(添加唯一约束),会自动在唯一字段创建唯一索引;
8.15.3 创建索引
8.15.3.1 创建唯一索引

–创建唯一索引:创建唯一索引的值不能重复
create unique index 索引名字 on 表名(列名)
create unique index index_tid on tb_textindex(tid);

8.15.3.2 创建普通索引

–创建唯一索引:不要求创建索引的列的值的唯一性
create index 索引名字 on 表名(列名)
create index index_tid on tb_textindex(tid);

8.15.3.3 创建组合索引

–创建唯一索引:不要求创建索引的列的值的唯一性
create index 索引名字 on 表名(列名1,列名2…)
create index index_tid on tb_textindex(tid, name);

8.15.3.4 创建全文索引
  • MySQL 5.6版本新增的索引，可以通过此索引进行全文检索操作，因为MySQL全文检索不支持中文，因此这个全文索引不被开发者关注，在应用开发中通常是通过搜索引擎(数据库中间件)实现全文检索。

–创建全文索引
create fulltext index 索引名字 on 表名(字段名);

8.15.4 索引使用

索引创建完成之后无需调用，当根据创建索引的列进行数据查询的时候，会自动使用索引。
组合索引需要根据创建索引的所有字段进行查询时触发。

  • 在命令行窗口中可以查看语句的查询规划:
    • explain select * from tb_textindex where tid = 1000\G;
8.15.5 查看索引

– 命令行

  • show create table 表名\G;
    • show create table tb_textindex\G;

– 查询数据表的索引
show indexes from tb_textindex;

– 查询索引
show key from tb_textindex;

8.15.6 删除索引
  • 索引是建立在表的字段上的，不同的表中可能会出现相同名称的索引，因此删除索引时需要指定表名
    • drop index 索引名字 on 表名;
    • drop index index_name on tb_textindex;
8.15.7 索引的使用总结
8.15.7.1 优点
  • 索引大大降低了数据库服务器在执行查询操作时扫描的数据，提高查询效率
  • 索引可以避免服务器进行排序、将随机IO变成顺序IO
8.15.7.2 缺点
  • 索引是根据数据表列的创建的，当数据表中数据发生DML操作时，索引也需要更新;
  • 索引文件也会占用磁盘空间;
8.15.7.3 注意事项
  • 数据表中数据不多时,全表扫描可能更快，不用使用索引;
  • 数据量很大但是DML操作很频繁时,不建议使用索引;
  • 不要在数据重复度高的列上创建索引(性别等);
  • 创建索引之后,要注意查询SQL语句的编写,避免索引失效。

8.16 数据库事务

8.16.1 数据库事务介绍
  • 我们把完成特定的业务的多个数据DML操作步骤称之为一个事务
  • 事务，就是完成同一个业务的多个DML操作

– 借书业务
– 操作1:在借书记录中添加记录
insert into records(snum, bid, borrow_num, is_return, borrow_date) values(‘1001’, 1, 1, 0, sysdate());
– 操作2:修改图书库存
update books set book_stock=book_stock-1 where book_id = 1;

转账业务:张三给李四转账
操作1:李四的账号+1000
操作2:张三的账号-1000

8.16.2 数据库事务特性

ACID特性，高频面试题

  • 原子性(Atomicity):一个事务中的DML操作，要么同时执行成功，要么同时执行失败
  • 一致性(Consistency):事务执行之前和事务执行之后，数据库中的数据是一致的，完整性和一致性不能被破坏
  • 隔离性(isotation):数据库允许多个事务同时执行(张三借Java书的同时允许李四借Java书)，多个并行的事物之间不能相互影响。
  • 持久性(Durability):事务完成之后，对数据的操作是永久的
8.16.3 MySQL事务管理
8.16.3.1 自动提交
  • 在MySQL中，默认DML指令的执行是自动提交的，当我们执行一个DML指令之后，自动同步到数据库中
8.16.3.2 事务管理

开启事务，就是关闭自动提交

  • 在开始事务第一个操作之前，执行start transaction(关闭自动提交)
  • 依次执行事务中的每个DML
  • 如果在执行的过程中的任何位置出现异常，则执行rollback(事务回滚(清除连接缓存中的操作,当前撤销事务已经执行的操作))
  • 如果事务中所有的DML操作都执行成功,则在的最后执行commit提交事务

– 开启事务(关闭自动提交–手动提交)
start transaction
insert into records(snum, bid, borrow_num, is_return, borrow_date) values(‘1004’, 3, 2, 0, sysdate());
– 出错
select aaa;
– 事务回滚(清除连接缓存中的操作,当前撤销事务已经执行的操作)
rollback;
update books set book_stock=book_stock-2 where book_id = 3;
– 提交事务(将连接缓存中的操作写入数据文件)
commit;

8.16.4 事务隔离级别

数据库允许多个事务并行，多个事务之间隔离的、相互独立的;如果事务之间不相互隔离并且操作统一数据时，可能胡导致数据的一致性被破坏

  • MySQL数据库事务隔离级别:
    • 读未提交(read uncommitted):T2可以读取T1执行但未提交的数据(可能会导致出现脏读)
      • 脏读:一个事务读取到了另一个事务中未提交的数据
    • 读已提交(read committed):T2只能读取T1已经提交的数据(避免了脏读，可能会导致不可重复读也叫虚读)
      • 不可重复读(虚读):在同一个事务中，两次查询操作读取到的数据不一致
      • 例如:T2进行第一次查询之后在第二次查询之前，T1修改并提交了数据，T2进行第二次查询时读取到的数据和第一次查询读取到的数据不一致
    • 可重复读(repeatable read):T2执行第一次查询之后，在事务结束之前之前事务不能修改(没说不能新增)对应的数据(避免了不可重复读(虚读)，但可能导致幻读)
      • 幻读:T2对数据表中的数据进行修改然后查询，在查询之前T1向数据表中新增了一条数据，就导致T2以为修改了所有数据，但却查询除了与修改不一致的数据(T1事务新增的数据)
    • 串行化(serializable):同时只允许一个事务对数据表进行操作(避免了脏读、虚读、幻读问题)
8.16.5 设置数据库事务隔离级别

我们可以通过设置数据库默认的事务隔离级别来控制事务之间的隔离性;
也可以通过客户端与数据库连接设置来设置事务间的隔离性(在应用程序中设置-Spring);
MySQL数据库默认的隔离级别为可重复读

  • 查看MySQL数据库默认的隔离级别

– 在MySQL 8.0.3之前
select @@tx_isolation;
– 在MySQL 8.0.3之后
select @@transaction_isolation;

  • 设置MySQL默认隔离级别
    • 语法:set session transaction isolation level 隔离级别;
    • 示例:set session transaction isolation level read committed;

9 数据库设计

MySQL数据库作为数据存储的介质为应用系统提供数据存储的服务，我们如何设计出合理的数据库、数据表以满足应用系统的数据存储需求呢?

  • 车库:是用来存放车辆的，车辆都需要划分车位，如果不划分车位，车子杂乱无章的存放可能会导致车辆堵塞，同时也可能造成场地的浪费–有限的场地可以停放最多的车辆，同时方便每一辆车的出入
  • 数据库:用来存放数据的，我们需要设计合理的数据表–能够完成数据的存储，同时能够方便的提取应用系统所需的数据
